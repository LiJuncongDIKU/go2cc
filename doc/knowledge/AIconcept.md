# 人工智能概念

## Vibe Coding
> Vibe Coding 是 2024 年末到 2025 年在开发者圈子火起来的一个新词。它描述了一种**“只提需求，不写代码”**的新型开发范式。

甚至一些公司的JD也开始写“Vibe Coding”，例如 Trea 的 solo 模式就是 Vibe Coding。但是目前看来，问题还是不少：
- 高质量提示词：虽然不像高级语言一样需要语法和逻辑细节，但提示词对于结果仍有很大的影响，编写高质量的提示词并不简单。
- 不可控的结果：以自适应布局为例，AI当然能做，但指定盒子的拉伸、主题颜色变量的方案等，需要特别完备的提示词或者出结果后微调。
- 代码质量：Vibe Coding 生成的代码质量参差不齐，微调的难度并不小，甚至要将某个小模块重建。
- 面向结果编程：Vibe Coding 强调的是“面向结果编程”，这意味着开发者需要关注结果，而不是逻辑细节。这可能埋下隐藏的BUG，或给调试带来一定难度，当然修BUG也交给AI，可能需要反复地跑，但是效率和开发体验就见仁见智了。

:::tip 发展 :seedling:
虽然上面说了一些AI高度参与开发后必然存在的问题，但实际上生态仍然不断迭代，自动化、针对开发使用的大模型等等，不断有新功能和逻辑解决上述问题，相信风潮会继续，并且AI发展越来越好。我从使用者的角度简单聊聊几个概念，尽量跟上时代的脚步。
:::

## LLM
LLM 是 Large Language Model 的缩写，指的是大型语言模型。它是一种基于 Transformer 架构的神经网络模型，能够理解和生成自然语言。
从chatGPT3开始，LLM的风潮开始刮起，但实际上是自然语言的问题揉碎了，推理结果，然后重组成新的句子。\
记得LLM发展的初期，其数学能力、理解能力都不是那么强，只是简单的文本生成，也有人称为“概率复读机。后面随着上面概述的流程不断完善，LLM内部添加了更多的步骤：
- 自洽性检查和错误修正：输出结果前，LLM会检查输出是否符合语法规则和逻辑一致性，若发现错误，会进行修正。
- 工具调用的能力
——*当然，这部分是拾人牙慧，大模型内部的算法没有系统性地学习* 🎃

## agent
在架构中，Agent 位于最顶层，它封装了 LLM、Rules、Skill 和 Memory。他将会做出最顶层的规划，例如问题解决的步骤，调用哪个工具、如何调用，如何整合结果等。

## skill
skill 相当于LLM的工具箱，LLM本质还是文本处理，但是它能够根据文本的上下文判断出需要调用的工具，例如调用API、js运行时工具等。\
这也使得LLM在实际应用中具有了更多的灵活性和功能。[vue-skill](https://github.com/vuejs-ai/skills) 是一个基于 Vue 3 的 LLM 工具库，社区影响力比较大，LLM能够通过类似的skill使得输出的结果更加符合预期。

## rules
rules 可以作为输出结果的过滤器，比如限制输出的格式、长度、内容，或者约定输出结果的口吻，比如“请用中文回答”、“请用正式的语言回答”，“请用xxx的口吻回答”等。

## memory
memory 是指 AI 模型的记忆能力。它包括了模型的短期记忆、长期记忆等。例如我们将文件内容作为短期记忆，模型在处理新的问题时，能够根据文件内容进行推理。将上下文（同一个任务内的历史问题和回答）作为长期记忆，模型在处理新的问题时，能够根据上下文进行推理。

## metadata
metadata 实际上是 memory、rules 的格式化组合，他将二者以json或其他特定方式进行组织，方便模型在处理新的问题时，能够根据 metadata 进行推理。不需要重复进行自然语言处理。

## Token
LLM 的计费与运算基本单位，一个 Token 约对应英文单词的 1/4（或中文字符 1 个），提示词和生成结果的长度均以 Token 计量。优化提示词、精简 Memory 可减少 Token 消耗，降低成本并提升推理速度。

## Temperature（温度值）
LLM 的核心参数，取值范围 0~1。值越高，输出结果越随机、有创意；值越低，输出越保守、精准。Vibe Coding 中，生成创意性原型可调高温度，生成核心业务代码需调低温度。

## RAG（Retrieval-Augmented Generation）
检索增强生成，是解决 LLM “知识过时”“幻觉” 问题的核心技术。通过先检索外部知识库（如企业文档、最新技术文档），再将检索结果作为 Memory 输入模型，让生成结果更贴合实际、减少错误。

## Fine-tuning（微调）
在通用 LLM 基础上，使用行业 / 场景专属数据进行二次训练，让模型更适配特定领域（如前端开发、金融风控）。相较于纯提示词工程，微调能让模型更稳定输出符合领域需求的结果。

## AI焦虑？
当我们了解上述AI项目的组成之后，我们知道AI即便是Solo模式，他仍然需要人类的参与、设计，例如节约Token、缩短推理时间、优化输出结果等。\
这也使得AI在实际应用中具有了更多的局限性，或者说使用AI进行开发并不是“有手就行”，需要一定的专业性和经验。\
在人类社会中，AI的定位必将是从整体上、大局上服务于人类，即使他确实强大，仍然需要**善于使用AI**的使用者。\
这便得出了网上常见的结论：
- 高效的工具可能导致岗位减少，但AI不会替代人类。
- AI作为新的生产工具，如果类比为电脑的发展，不使用AI的人，类似当下社会不少见的*电脑文盲*。
:::danger 🙊 关于整个行业的大胆想象和推论
之前马斯克说应该跳过开发工程的阶段，直接开发二进制产品。倒是启发了我一些思考：**中间产物**。
以传统开发团队来说，原型图、UI 图、需求文档、UML用例、模块泳道图等都是中间产物，它们将开发团队的各个岗位连接起来，规范地留下了设计系统架构、验证功能需求等。\
正如现如今使用的*代码*，是编译器和开发者同时阅读。在AI工具尝试颠覆传统开发模式的时候，应该存在类似的中间产物，而其阅读者同时是AI模型和人类。而现在*代码*（高级语言）直接充当这个角色恐怕不妥——Skill更像是插件，而不是一个成体系的、能够跨域前后端各类产品和定位的中间产物。换句话说，通用大模型的场景恐怕不是王道，AI应用接下来会朝着精细赛道前进。吧~
:::
