# 音视频处理
音视频本来可以作为文件处理，但不可能文件下载完再播放，因而引入**流媒体**的概念。
:::info 流媒体
流媒体（Streaming Media）是一种通过互联网即时传输音视频数据、无需等待完全下载即可开始播放的技术。它将文件分段压缩并通过网络实时传送至用户设备，广泛应用于Netflix、YouTube、Spotify等视频/音乐平台，实现了随点随看、直播及互动等无缝体验。——维基百科
:::

音视频从场景上基本分为三种：
- 点播模式：类似视频网站
- 直播模式：类似实时监控、直播平台
- 实时互动模式：类似视频会议，复杂度在于多对多的互动。

## 点播模式
先有视频，后播放的场景，特点是：需要跳转、提前缓存、延迟不敏感。通常使用 HLS 或 DASH：视频被切成固定长度的分片（Segments），播放器（客户端）根据配置获取不同码率、语言的视频或音频分片。
#### HLS 和 DASH
这两个是目前比较常用的流媒体协议，其职责是把媒体文件切片成小片，通过索引文件告诉客户端如何播放。
| 维度       | **HLS (HTTP Live Streaming)**                  | **DASH (Dynamic Adaptive Streaming over HTTP)**               |
| :--------- | :--------------------------------------------- | :------------------------------------------------------------ |
| 主导公司   | Apple（绝对话语权）                            | MPEG 组织 (动态影像专家小组，Google、Microsoft、Adobe 等推动) |
| 切片格式   | 传统为 `.ts`，现全面支持 .m4s (CMAF)           | 原生支持 `.m4s` / `.mp4` (CMAF)                               |
| 索引文件   | `.m3u8` (基于文本的播放列表)                   | `.mpd` (基于 XML 的媒体呈现描述)                              |
| 方案       | 默认ts格式将音视频切片到一起，对应m3u8索引文件 | m4s格式将音视频分开切片，对应mpd索引文件                      |
| 自适应码率 | 支持（由 Apple 定义切换逻辑）                  | 极强（支持更精细的码率、多语言、多视角）                      |
| 视频编码   | H.264/AVC（H.265/HEVC、AV1）                   | H.264/AVC（H.265/HEVC、AV1、VP8/9）                           |
| 音频编码   | AAC、MP3、AC3/E-AC3                            | AAC、Opus、MP3、AC3                                           |

> 大多数浏览器的`video`标签都能直接播放：编码组合：H.264/AVC + AAC

## 直播模式
场景为：一个主播实时推流，多个观众实时拉流播放。特点是：
- 延迟敏感：观众需要实时观看，不能等待太长时间。
- 带宽要求高：主播需要传输高码率的视频流，观众需要接收并解码。
```
主播端采集
   ↓
编码（H.264 + AAC）
   ↓
推流（RTMP / WebRTC）
   ↓
服务器实时切片（生成 HLS / DASH / FLV 流）
   ↓
CDN 分发
   ↓
播放器拉流 N个
   ↓
解封装 → 取H.264 → 解码 → 播放
```

## 实时互动模式
场景为多个用户同时作为主播和观众，需要实现：
- 主播可以实时推流
- 观众可以实时拉流并播放
- 主播和观众之间可以互动（如聊天、点赞等）

| 架构 | 说明 |
| :--- | :--- |
| Mesh | 参与者两两直接连接（适合 3 人以下）。 |
| MCU | 服务器将所有人的画面混成一路流发给所有人（计算压力在服务端）。 |
| SFU | 服务器只负责数据包的转发，不解码，解封装、解码由客户端负责。 |
